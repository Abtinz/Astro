# floor-plan-3d Structure and Architecture

This document explains the `Astro/floor-plan-3d` app structure and how its generation pipeline works.

## 1) Purpose

`floor-plan-3d` is the interactive app that converts a 2D floor plan image into:
- A photorealistic generated render
- A downloadable interactive 3D HTML scene

## 2) Tech Stack

- React 19 + TypeScript
- Vite
- `@google/genai` for Gemini calls
- Three.js scene code generated by model output (then post-processed)

## 4) Core File Responsibilities

### `src/main.tsx`
- React entrypoint.
- Mounts `<App />` into `#root`.

### `src/App.tsx`
- Main orchestration/state machine for the full user journey.
- Owns app statuses:
  - `idle`
  - `generating_render`
  - `extracting_walls`
  - `generating_base_scene`
  - `generating_furniture`
  - `validating_voxels`
  - `error`
- Owns view modes:
  - `upload`
  - `render`
  - `voxel`
- Coordinates all Gemini pipeline stages and passes thought/progress text to UI.
- Handles downloads for generated image and final HTML scene.

### `src/components/UploadZone.tsx`
- Drag/drop and click-based file intake.
- MIME allowlist validation (`png`, `jpeg`, `webp`, `heic`, `heif`).
- Converts file to base64 data URL for pipeline input.

### `src/components/Stepper.tsx`
- 3-step visual progress indicator:
  1. Upload
  2. 3D Render
  3. 3D Scene

### `src/components/Viewer.tsx`
- Displays one of:
  - Empty state
  - Image preview (`<img>`)
  - Voxel scene preview (`<iframe srcDoc>`)
- Uses iframe sandbox for generated scene execution.

### `src/components/LoadingOverlay.tsx`
- Shows generation phase text, phase label, pseudo-progress, and streamed thought snippet.

### `src/services/gemini.ts`
- Encapsulates all LLM/model interaction with Gemini.
- Exposes pipeline functions:
  1. `generateFloorPlanRender`
  2. `extractWallJSON`
  3. `generateBaseScene`
  4. `generateFurnitureScene`
  5. `fixVoxelScene`
- Supports streaming responses and thought fragments for progressive UI feedback.

### `src/utils/html.ts`
Post-processing and parsing utilities for model output:
- Extraction:
  - `extractHtmlFromText`
  - `extractJsonFromText`
- Scene transformations:
  - `hideBodyText`
  - `zoomCamera`
  - `injectWASDControls`
  - `injectCityEnvironment`
  - `fixZFighting`

Applied transform chain in `App.tsx`:
`hideBodyText -> zoomCamera -> injectWASDControls -> injectCityEnvironment -> fixZFighting`

## 5) Runtime Data Flow

```text
Upload image
  -> base64 data URL
  -> generateFloorPlanRender
     -> rendered image
  -> extractWallJSON
     -> structural JSON
  -> generateBaseScene
     -> base architectural HTML scene
  -> generateFurnitureScene
     -> furnished HTML scene
  -> fixVoxelScene
     -> validated/corrected HTML scene
  -> html.ts transform chain
  -> Viewer iframe preview + HTML download
```

## 6) Config and Environment

### `vite.config.ts`
- Dev server port: `3001`
- React plugin enabled.
- Defines `process.env.API_KEY` from `GEMINI_API_KEY` (via `loadEnv`).
- Adds alias `@ -> src`.

### `package.json`
Commands:
- `npm run dev`
- `npm run build`
- `npm run preview`

## 7) Security/Operational Notes

- Gemini API usage is currently client-side (`process.env.API_KEY` injected at build/runtime).
- For production hardening, move model calls behind a backend proxy/service to avoid exposing sensitive API keys.
- Generated scene HTML executes in an iframe with sandbox flags; keep sandbox policy tight if capabilities expand.

## 8) Quick Maintenance Guide

1. UI flow or button behavior: edit `src/App.tsx`.
2. Upload/preview issues: inspect `UploadZone.tsx` and `Viewer.tsx`.
3. Model prompt or generation quality changes: edit `src/services/gemini.ts`.
4. 3D output rendering artifacts (camera, controls, z-fighting): edit `src/utils/html.ts`.
5. App styling/theme: edit `src/App.css`.
